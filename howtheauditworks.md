The Audit Categories
1. Structural Parsability (can the agent even read the page?)
This is foundational. You'd measure how well the page converts from raw HTML into something an LLM can reason about. Things like: DOM depth and nesting complexity, ratio of meaningful content to boilerplate/chrome, how much content is rendered client-side (invisible to simple fetch), whether text is trapped in images/SVGs/canvas elements, and iframe nesting depth. A page that's 90% navigation chrome and 10% actual content is expensive and noisy for an agent.
2. Semantic Clarity (does the agent understand what it's looking at?)
This goes beyond structure into meaning. Does the page use semantic HTML (article, nav, main, aside) or is it all divs? Are headings hierarchical and descriptive? Do buttons and links have meaningful labels vs. "Click here" or icon-only? Are form fields properly labeled with associated <label> elements? Is there ARIA where it matters — not for humans this time, but because agents using accessibility trees rely on it heavily. You could score this as an "Agent Comprehension Index."
3. Interaction Surface (can the agent actually do things?)
For agentic use cases (booking, filling forms, navigating flows), you'd audit: how many clicks to reach key actions, whether forms have predictable field types and validation feedback, whether success/error states are machine-readable (not just a color change), multi-step flow linearity (can an agent reason about "what's next?"), and whether modals, dropdowns, and dynamic UI elements are accessible via the DOM or require visual reasoning. This is where most sites silently break agents.
4. Anti-Agent Friction Score
A critical and unique category. You'd detect: CAPTCHAs and challenge walls, aggressive bot fingerprinting (Cloudflare, Akamai, etc.), rate limiting behavior and thresholds, cookie consent walls that block content, login walls and authentication complexity, and geo-blocking or user-agent filtering. You wouldn't help people bypass these — you'd help site owners understand the friction their legitimate agent users face.
5. Data Extraction Quality
How reliably can structured data be pulled? Does the site use schema.org / JSON-LD / microdata? Are prices, dates, addresses, and product info in parseable formats? Do tables use proper <table> markup or are they CSS grid layouts? Is pagination machine-navigable? Are APIs documented and available as alternatives to scraping? You'd compare "what a human sees" versus "what an agent can extract" and score the gap.
6. Token Economics (the performance angle)
This is the agent equivalent of Lighthouse's performance score. How many tokens does it cost to represent this page? What's the content-to-noise ratio after HTML-to-markdown conversion? How much redundant content appears across pages (headers, footers, sidebars)? What's the cost delta between a full page fetch vs. an ideal API call? You could literally show: "This page costs $0.03 per agent visit with Sonnet, but only $0.004 if you added a JSON-LD layer." That's a number enterprises understand.
7. Temporal Stability (will it break tomorrow?)
Agents that rely on DOM structure are fragile. You'd measure: how frequently CSS class names and DOM structure change, whether the site uses generated/hashed class names (CSS modules, Tailwind JIT), consistency of element selectors across deploys, and A/B testing interference (does the page change randomly?). A stability score over time — like "this site's checkout flow has changed structure 4 times in 30 days" — is incredibly valuable for anyone building agent workflows.
8. Multi-Modal Readiness
As agents increasingly use vision: are key actions visually distinguishable? Is there sufficient contrast and spatial separation between interactive elements? Are there visual cues that map to function (a button looks like a button)? How does the page render at the viewport sizes agents typically use?
